Why this page exists

"PrimeDox will wear many names, A to Z, but will always be PrimeDox." This page is that instruction made literal: engine/rl-rule-engine.js — the exact file backing Timmy's leaderboard — is imported here too, just constructed as new PrimeDoxRuleEngine('PrimeDox AI', 'document-suggestion-optimization', ...) instead of 'Timmy'. One scorer, two brands, separate policy storage so they never collide.

The outcome vocabulary is generic on purpose — blocked_threat / false_positive / missed_threat / true_negative aren't security-specific, they're just "correctly acted / wrongly acted / wrongly silent / correctly silent." Here that maps to: a document-type or citation suggestion the user accepted (+1), one they dismissed as irrelevant (-2), a moment PrimeDox should have suggested something and didn't (-3), and correctly staying quiet when nothing applied (+0.1).

This is a static rule scorer, not a self-evolving AI, and it is not wired to PrimeDox AI's actual system prompt (primedox-ai-backend/main.py) at all — that prompt stays a static string a human edits directly. When a rule's score crosses a lifecycle threshold here, the engine raises a pending suggestion in the "Needs your approval" panel below and stops. Derek reviews and clicks Approve or Dismiss — nothing promotes, demotes, or disables itself. The lifecycle ladder (sandbox → active → core, or anything → disabled) and the human-review safety valve on Core rules both come along for free, the same as on Timmy's dashboard.
